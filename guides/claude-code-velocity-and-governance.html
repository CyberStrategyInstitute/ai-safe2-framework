Multi-Agent Execution

Stop working sequentially. The architecture was built for parallel — most developers aren’t using it.

/batch — Fan Out to Hundreds of Agents

What it does: Interviews you about the task, then fans out to as many parallel worktree agents as needed — dozens, hundreds, even thousands for large migrations. Each agent gets its own isolated branch, working simultaneously.

Why it’s nearly free at scale: When Claude Code forks a subagent, it creates a byte-identical copy of the parent context. The API caches this. Five agents in parallel costs barely more than one sequentially. The source code confirms this architecture explicitly — it was designed for it.

Example: /batch for parallel code migration 
# Start batch mode — Claude interviews you, then fans out
/batch

# Claude spawns parallel agents like:
#  Agent 1 → auth module refactor     (worktree: feature/auth-refactor)
#  Agent 2 → test coverage pass       (worktree: feature/test-coverage)
#  Agent 3 → dependency updates       (worktree: feature/deps-update)
#  Agent 4 → docs sync               (worktree: feature/docs-sync)
#  Agent 5 → security audit          (worktree: feature/sec-audit)
# All run simultaneously. All share the prompt cache.
⚠ Risk: Blast Radius Scales With Agent Count
When you fan out to 100 agents, any single compromised subagent’s blast radius becomes instantaneous. Subagents inherit the parent’s permission state including any bypassed flags. Map what each agent can reach before scaling. See the Sovereign Runtime package for subagent permission control.

/branch — Fork Without Losing Context

What it does: Forks a running session into two parallel threads. Deep into a hard problem, want to explore an alternative? Branch it. The existing session keeps running. The fork explores the alternative. Compare outcomes.

CLI equivalent: claude --resume --fork-session — sessions persist at ~/.claude/projects/{hash}/{sessionId}.jsonl. Always use --continue by default. Stop starting fresh every session.

claude -w — Worktrees for True Isolation

Starts a new session in a fresh git worktree. Run dozens of Claude instances simultaneously in the same repo, each with its own isolated branch. Combined with /batch, this is the mechanical foundation of parallel execution at scale.

Parallel worktree setup 
# Spawn isolated agent in its own git worktree
claude -w

# Each -w agent gets:
#  - Its own git worktree + isolated branch
#  - Independent file system state
#  - Shared prompt cache (cost-efficient)

# Always resume sessions — never start fresh
claude --continue

--agent — Scoped Roles with Custom Tools

Loads a custom system prompt and toolset from .claude/agents/. A reviewer agent with read-only tools is a fundamentally different risk context than a deployer with write access — this is least-privilege by architecture, not by hoping.

--agent reviewer
Code Review Agent
Read-only. Suggests changes, never writes, executes, or modifies. Safe for untrusted code review.
LOW RISK
--agent researcher
Research Agent
Web search and read only. No file writes, no bash. Safe for information gathering tasks.
LOW RISK
--agent deployer
Deploy Agent
Constrained write access to deployment configs only. Explicit deny on source code edits.
MED — configure carefully
.claude/agents/reviewer.json 
{
  "name": "reviewer",
  "description": "Code review — read-only, no execution",
  "system": "You are a senior code reviewer. Read and suggest. Never write, execute, or modify.",
  "tools": ["Read", "Search", "Grep", "ListFiles"],
  "permissions": { "deny": ["Write(**)", "Edit(**)", "Bash(*)"] }
}

Permission System — Configure It Once

5-level cascade: policy → flag → local → project → user. Every permission click is time wasted. Set glob patterns once in settings.json and never click again.

~/.claude/settings.json — recommended baseline 
{
  "permissions": {
    "allow": [
      "Bash(npm *)", "Bash(git *)", "Bash(make *)",
      "Edit(src/**)", "Write(src/**)", "Read(**)"
    ],
    "deny": [
      "Bash(rm -rf *)", "Bash(curl * | bash)",
      "Write(/etc/**)", "Write(~/.ssh/**)"
    ]
  }
}
